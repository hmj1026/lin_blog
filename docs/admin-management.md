# 後台管理操作與資料治理指南

本文件說明 Lin Blog 後台的日常管理流程、預設角色權限、文章統計定義，以及活動紀錄的隱私、保存與回滾原則。實際授權仍以伺服器 session 與 API 權限檢查為準；側邊欄隱藏入口只是操作提示，不是安全邊界。

## 日常管理流程

### 儀表板

- 儀表板依登入者權限顯示草稿、排程、近期更新、文章成效與快捷操作；無權限的捷徑不會出現。
- 統計讀取失敗時使用頁面上的「重試」，不要以重新送出其他表單代替重試。
- 「成長」與「衰退」以同長前期比較；前期為 0 且本期有瀏覽時顯示為新流量，不計成無限百分比。

### 文章與編輯器

1. 在「文章列表」以關鍵字、狀態、分類、標籤、精選與排序縮小結果；篩選、排序與頁碼會保留在 URL，可安全分享或重新整理。
2. 批次操作會逐篇回報成功與失敗；變更篩選條件後，先前選取會被清除，避免對看不見的文章操作。
3. 軟刪除文章可從垃圾桶復原。永久性或覆蓋操作會先顯示影響範圍，確認後才執行。
4. 編輯器只在內容變更時自動儲存，且不會自行改變發布狀態。若出現版本衝突，停止編輯並重新載入最新版本後再合併內容。
5. 發布前確認欄位錯誤、SEO 預覽、閱讀時間、分類與標籤。離頁警告表示仍有尚未保存的變更。

### 分類、標籤與媒體

- 分類與標籤列表支援搜尋、排序、軟刪除、復原與安全合併。合併前確認來源與目的項目，以及受影響文章數。
- 分類導覽只在「分類管理」維護，避免站點設定出現第二個寫入來源。
- 媒體庫以 URL 保存搜尋、類型與分頁。刪除前會檢查封面、OG、Hero、文章 HTML 與 Raw HTML 引用；解析結果不確定時系統會安全阻擋。
- 媒體刪除說明中的「文章內文引用未檢查」警語只適用無法可靠解析的舊資料；應先人工確認再處理，不要繞過阻擋。

### 訂閱者、使用者、角色與站點設定

- 訂閱者頁可搜尋、分頁與複製單筆 email；禁止批次複製、匯出與群發，以維持個資暴露邊界。
- 使用者的角色調整、停用與密碼重設是不同操作。系統會阻擋自我停用與停用最後一位管理者。
- 刪除角色前須先重新指派其啟用使用者。角色頁可複製既有範本，並會阻擋缺少相依權限的組合。
- 站點設定依分區保存，只傳送已變更欄位；部分失敗時保留未儲存輸入。可先以桌面或手機預覽確認，再重新送出失敗分區。
- 「關於我」管理入口依 `settings:manage` 顯示；「前台已啟用／未啟用」只描述公開顯示狀態，不影響管理權限。

## 預設角色權限矩陣

ADMIN 預設取得全部權限；EDITOR 採明確白名單。自訂角色可在「角色權限」調整，但除 `admin:access` 本身外，各後台能力都必須同時包含 `admin:access`；`analytics:view_sensitive` 另依賴 `analytics:view`。

| 權限 | 能力 | ADMIN | EDITOR |
|---|---|:---:|:---:|
| `admin:access` | 進入後台與儀表板 | ✓ | ✓ |
| `posts:write` | 文章、關於內容與匯入／匯出 | ✓ | ✓ |
| `uploads:write` | 上傳與媒體庫 | ✓ | ✓ |
| `analytics:view` | 彙總文章統計 | ✓ | ✓ |
| `analytics:view_sensitive` | 原始事件、遮罩 IP 與 UA 稽核 | ✓ | — |
| `categories:manage` | 分類管理 | ✓ | — |
| `tags:manage` | 標籤管理 | ✓ | — |
| `users:manage` | 使用者管理 | ✓ | — |
| `roles:manage` | 角色與權限管理 | ✓ | — |
| `settings:manage` | 站點設定與關於我設定 | ✓ | — |
| `subscribers:view` | 訂閱者名單與單筆 email 複製 | ✓ | — |
| `audit:view` | 高風險管理活動紀錄 | ✓ | — |

權限變更後，既有 session 不會即時重建；受影響使用者必須重新登入。無權限直接造訪管理頁時會看到 403，API 也會獨立拒絕請求。

## 文章統計定義

- **日界線與範圍**：所有日期依 `Asia/Taipei` 日曆日切分，包含今天；`days` 限制為 1–90 天。沒有事件的日期補 0。
- **本期與前期**：前期是緊接本期之前、長度相同且不重疊的區間。
- **瀏覽數**：區間內符合條件的完整資料庫事件數，不再截斷為前 5,000 筆。
- **每日平均**：本期瀏覽數除以所選日數；零事件日期仍計入分母。
- **變化率**：`(本期 - 前期) / 前期 × 100%`，四捨五入為整數。兩期都是 0 時為 0%；前期 0、本期大於 0 時標記為新流量。
- **文章篩選**：可依分類、標籤與發布日期篩選，篩選套用於彙總、排行與前期比較。
- **來源占比**：以該範圍來源事件數除以總來源事件數。來源分類的單一規則如下：
  - 站內：相對路徑或本站 host。
  - 自然搜尋：Google、Bing、Yahoo、DuckDuckGo、Baidu 等已知搜尋來源。
  - 社群：Facebook、Instagram、LinkedIn、X/Twitter、Threads、YouTube 等已知社群來源。
  - 外部推薦：可解析、但不屬於前三類的外部 host。
  - 直接／未知：空值或無法解析的 referer。
- **敏感事件**：一般統計只呈現彙總結果。原始事件需要 `analytics:view_sensitive`；IP 預設遮罩，完整識別資訊不可寫入一般活動紀錄。

## 活動紀錄、隱私與保存

- 高風險操作成功後才寫入活動紀錄，包含 actor、action、resource type/id、時間與經白名單縮減的摘要。活動紀錄失敗不會把已成功的主要操作回報為失敗，但伺服器會留下不含秘密的維運警告。
- 摘要只允許影響數量、變更欄位、前後角色／狀態、範圍與引用識別等有界欄位；不得包含密碼、token、session、完整文章內容、完整 IP 或任意 request body。
- `changedFields` 只記錄欄位名稱，不記錄秘密值。字串與陣列另有限長，避免把大量內容變相寫入紀錄。
- 活動紀錄保存 365 天；列表查詢也以 365 天為最早邊界，每頁最多 100 筆。系統清理超過期限的資料。
- 僅具 `audit:view` 的角色可查看活動紀錄。篩選可使用日期、actor 與 resource；查閱權不代表可執行原操作。

## 部署與回滾

本變更新增 `AuditEvent` table、索引與 `audit:view` 權限，屬 additive migration。部署時先備份資料庫，再執行標準 `npx prisma migrate deploy`，並使用既有 seed／初始化流程以冪等方式補齊權限。

預設回滾方式：

1. 停止部署新版本並回退至上一個應用程式 release。
2. 驗證登入、文章、媒體與既有管理流程。舊版可忽略新增的 table 與權限，因此不需要回滾資料庫。
3. 保留 `AuditEvent` 至 365 天保存期限屆滿或完成資料治理審查；保留 additive table 不影響舊版應用程式。
4. 只有在確認沒有任何應用版本或角色仍引用時，才可另案移除 `audit:view` 的角色關聯與權限目錄資料。

移除 `AuditEvent` table 會永久刪除稽核證據，不屬於一般應用回滾。若法規或資料治理決策要求移除，必須先完成備份、保存責任交接、目標資料庫確認與人工核准，再由資料庫管理者另行撰寫並演練 destructive migration；不得直接在正式環境手動執行 `DROP TABLE` 或刪除 Prisma migration history。

## 上線後驗證

- 以 ADMIN、EDITOR 與自訂分析唯讀角色重新登入，確認側邊欄、403 與 API 權限一致。
- 在 375、768、1024、1440px 檢查表格、卡片、對話框、焦點與水平捲動；以鍵盤完成搜尋、文章建立、確認與手機導覽。
- 建立一筆非正式高風險測試操作，確認主要操作成功、活動紀錄可依 actor/resource 找到，摘要未出現密碼、token、內容或完整 IP。
- 對固定日期範圍比對本期、同長前期、零值日期與來源合計；來源數合計應等於來源摘要的總事件數。
